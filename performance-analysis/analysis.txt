TITLE: Improving the use of CPU micro-op execution ports via code interleaving.

PART I.
Introduction.

While analyzing the performance of my side project's communication server on
GNU/Linux using Top-down microarchitecture analysis methodology, I saw that the
Core-Bound metric (a sub-bucket of Backend-bound) was consistently being
reported to be nearly 10%, with tma_ports_utilization matching that percentage.

I obtained this using the Perf tool:

perf stat --topdown -- bin/rosetta-server
perf stat -M tma_core_bound,tma_memory_bound -- bin/rosetta-server
perf stat -M tma_ports_utilization -- bin/rosetta-server

To find out which function was contributing the most to this, I used:

perf record -e cpu_core/exe_activity.1_ports_util/ -g -- bin/rosetta-server
perf report

This reported that 61% of cycles in which the pipeline was stalled due to poor
execution port utilization in the CPU backend (as calculated by the Top-down
analysis methodology) was attributed to my Montgomery Modular Multiplication
function, montgomery_mul(), under the "self" column.

This is also the function that takes up the most time of anything else in the
running communication server, since it has to validate a cryptographic signaure
on each message it receives, which involves the following modular powering:

R1 = (G^s) mod M
R2 = (A^e) mod M

G, A and M are 3072-bit unsigned Big Integers (implemented by myself in C),
while the powers e and s are 320-bit, so this is nothing short of a gargantuan
calculation. The server has to perform it all the time, as each connected
client sends poll requests 10 times per second, each of which contain a
signature that must be validated by the server.

The montgomery powering code spends most of its time in montgomery_mul() calls,
so I directed my focus to montgomery multiplication and the way I implement it.

The vast majority of its work is, in turn, done in the following inner loop:

static inline
__attribute__((always_inline))
void mont_mul_inner_loop(      uint64_t* const __restrict__ R_limbs,
                         const uint64_t* const __restrict__ N_limbs,
                         const uint64_t* const __restrict__ X_limbs,
                         const uint64_t* const __restrict__ Y_limbs,
                         const uint64_t i,
                         const unsigned long long q)
{
    uint8_t C;
    uint8_t D;
    unsigned long long  Ul;
    unsigned long long  Uh;
    unsigned long long  Vl;
    unsigned long long  Vh;
    unsigned long long  W;

    /* T, a 3-limb variable, resides in R->bits buffer for optimization:
     * T = (unsigned long long*)(R->bits + ((MONT_L + 1) * MONT_LIMB_SIZ));
     */
    #define T  ( (unsigned long long*)(R_limbs + (MONT_L + 1)) )
    for(u64 j = 1; j < MONT_L; ++j){
        Ul   = _mulx_u64(q, N_limbs[j], &Uh);
        Vl   = _mulx_u64(Y_limbs[i], X_limbs[j], &Vh);
        C    = _addcarryx_u64((u8)0, Ul, R_limbs[j], &Ul);
        Uh  += (u64)C;
        D    = _addcarryx_u64((u8)0, Vl, T[1], &Vl);
        C    = _addcarryx_u64((u8)0, Ul, Vl, T);
        D    = _addcarryx_u64(D, Uh, Vh, &W);
        C    = _addcarryx_u64(C, W, T[2], T + 1);
        T[2] = (u64)C + (u64)D;
        R_limbs[j-1] = *T;
    }
    #undef T
}

I had already factored the inner loop out into a function, so I could give the
compiler's memory aliasing analysis clear hints that I guarantee the pointers
to the actual bits of the Big Integers (to their limbs, a limb being 64 bits)
are guaranteed to be the only pointers used to access that memory buffer in
the pointer's lifetime, so the compiler can more aggressively optimize the loop
with its own optimization passes. Of course, this guarantee is true by design
in my system. I was not able to do this by simply opening a new local scope
with a { } block in montgomery_mul() and declaring the pointers as __restrict__
there. GCC seems to prefer this info be given to it on the function argument
level. I did this optimization separately from my interleaving effort.

For the BigInt sizes I stated, the two Montgomery powering calls run a combined
total of 2,165,760 iterations of this inner loop. To keep this as short as I
can, I won't give detailed steps/explanation to parts that I deem can be taken
for granted, such as how I get this total number of inner loop iterations here.

I theorized that the dependencies in Montgomery Multiplication itself could
be leading to not utilizing the CPU backend's execution ports optimally,
despite the Reservation Station having micro-ops ready to feed the ports.

We can clearly see such a dependency here:

D = _addcarryx_u64((u8)0, Vl, T[1], &Vl);
C = _addcarryx_u64((u8)0, Ul, Vl, T);
D = _addcarryx_u64(D, Uh, Vh, &W);
C = _addcarryx_u64(C, W, T[2], T + 1);

The first call stores its result in Vl and any produced carry in D, then the
next call immediately uses Vl as input. The 4th call uses results produced by
the second and third calls (64-bit variable W and the carry C).

Therefore, I experimented with an interleaved version of Montgomery powering
that can process two independent powering streams at once, and thus get the
chance to call an interleaved montgomery_mul() that would operate on both
operand sets in the same function, with the most impactful part of this
being the interleaved inner loop for the new dual_montgomery_mul(). I have
interleaved the code for the two independent multiplication streams in the
inner loop by simply doing an iteration on one operand set and then, before
going to the next iteration, doing the same steps for the second operand set.

In the end, it proved to be a significant performance win.

It decreased the time taken to perform these two montgomery modular powering
calculations by 10%. These two calculations are by far the largest part of
validating a signature, which, in turn, is what the communication server spends
the vast majority of its time doing. I explain exactly how I measured the
average latency these two modular powering operations take in the next section.

Average latency before interleaving the two: 4703 microseconds.
Average latency after  interleaving the two: 4213 microseconds.

This is all using my own C implementation.
Running on a 13th Gen Intel Core i7 CPU, singlethreaded, on a GNU/Linux system
locally installed, kernel 6.19, compiled by GCC 15.2.1 with -O3 -march=native.

PART II.
Measuring latency.

I surrounded the two montgomery powering calls with gettimeofday() and took
the difference of the two recorded times to obtain each latency reading. Since
each reading is upwards of 4000 microseconds, gettimeofday() is safe to use
here without itself messing with the measured latency readings, as this call is
implemented in modern GNU/Linux systems to simply read a piece of memory mapped
by the kernel for each process, readily available without switching to kernel
space at all, thus we obtain the current time in much less than 1 microsecond.

Once I've collected 500 latency readings from the running communication server,
they get dumped to a file with fwrite(). A small python program then computes a
stabilized average latency from these 500 readings in the following way:

First, eliminate outliers:

1. Sort the latency readings in ascending order. Split into 2 halves.
2. middle-indexed reading of first half (median) = Q1. The 25th percentile.
3. middle-indexed reading of other half (median) = Q3. The 75th percentile.
4. Interquartile range IQR = Q3 - Q1.
5. LowerFence LF = Q1 - (1.5 * IQR).
6. UpperFence UF = Q3 + (1.5 * IQR).
7. Outliers are all readings LESS THAN LowerFence OR GREATER THAN UpperFence.

With the outliers gone, we take the average of the remaining readings.

Repeat this process of collecting batches of 500 latency readings, removing
outliers and noting their average (sum divided by count) 128 times. This gives
us tens of thousands of individual latency readings to work with.

With 128 stabilized averages, we compute the final average latency.

Using this method, I consistently get final averages within 2 microseconds of
the median final average. That is solid, given my 4000+ microsecond readings.

PART III.
Proving my hypothesis for why we get this speed-up and explaining it from a CPU
micro-architecture point of view.

Interleaved version is 10% faster because the Reservation Station is now able
to feed the backend's execution ports with micro-ops from instructions that
belong to the second chain of Montgomery multiplication, out of order, while
waiting on Chain 1's dependencies to give their results to stalled uOPs in RS.

In the non-interleaved version, the Reservation Station had instructions fed to
it by the CPU Frontend but it was not able to feed their micro-ops to execution
ports because the instructions in the RS had dependencies on ones that were
running already in the ports. In other words, I improved instruction-level
parallelism usage and made better use of the Out-of-Order capabilities of
modern CPU microarchitecture by moving the 2nd Montgomery Multiplication chain
to be interleaved with the code for the first, instead of one after another.

Interleaving two streams in one function does increase register pressure and
makes the compiler spill registers more often, but the hot inner loop is still
small enough that everything fits in L1 cache and these trips to L1 cache are
masked by out-of-order execution of the other Multiply stream's micro-ops.

Simulating the microarchitecture pipeline in LLVM Machine Code Analyzer to run
the instructions for the inner loop (which is >99% of the work) of both versions
of Montgomery multiplication gave insights into important metrics:

I. How many cycles per iteration each execution port was being used in.

	 and compute total cycles per iteration, then percentage utilization of

	 These percentages showed that the interleaved version BETTER utilizes the
	 CPU backend's execution ports overall:

II. Number of Assembly instructions occupying the CPU backend in parallel.

From the pipeline simulation Timeline view, I can see that the interleaved
version gets more instructions residing in the CPU backend at various stages
in their execution (waiting in the Reservation Station to be fed to execution
port(s), being executed in port(s), or waiting in the Reorder Buffer, with their
result visible to the pipeline, to be retired in-order). Taking the length in
cycles of one of the longest-residing instructions (it's a recurring pattern,
the same instruction keeps being the longest residing one), I can see:

NOTE: Instructions per iteration start at index 0.

Interleaved version:

----------------------------------------------------------------------------
D===================eeeeeeeeeeeeER  mov qword ptr [rdx + 400], rsi
                ...
                                D=eE-------------------------R  cmp rax, 48
----------------------------------------------------------------------------

Iteration 9, instruction 30 and iteration 10, instruction 72 are what is
displayed above, with 73 instructions per loop iteration: 117 instructions
being processed by the CPU backend in parallel, in the best case.

Non-interleaved version:

----------------------------------------------------------------------------
D==============eeeeeeeeeeeeER  mov qword ptr [rax + 400], rcx
                ...
                           D======E------------------R  movzx  esi, sil
----------------------------------------------------------------------------

Iteration 21, instruction 31 and iteration 24, instruction 17 are what is
displayed above, with 40 instructions per loop iteration: 108 instructions
being processed by the CPU backend in parallel, in the best case.

Timeline diagram context:

- Each character from D to R is one cycle.
- D: Instruction was given to the Reservation Station by the CPU Frontend.
- =: Instruction waits in RS, can't yet go to its required execution port(s).
- e: Instruction's micro-ops are being worked on by execution port(s).
- E: Instruction's result is ready and visible to the pipeline at this cycle.
- R: Instruction is retired in-order from the Reorder Buffer at this cycle.

As long as we stay within the limits of the Reorder Buffer's number of slots,
which we are here, this is a clear win for performance. It shows that the
interleaved version is better tapping into the benefits of the highly parallel,
out-of-order design of the microarchitecture of CPUs. It clearly indicates the
backend's execution ports are in fact being better utilized by the interleaved
version, which was my hypothesis as to why it's faster in the first place.

It's possible to reach a number of assembly instructions residing in the
backend in parallel that is TOO HIGH. When the CPU Frontend wants to feed
the CPU Backend an instruction, that instruction needs:

- A free slot in the Reservation Station, to wait for its operands if needed.
- A free slot in the Reorder Buffer (ROB), so it can be retired in-order.

If either of these limits is hit, the CPU Frontend simply stops feeding
instructions to the CPU Backend, stalling the pipeline. This can then lead
to decoders not having free space to place decoded uOPs of instructions in,
plus the instruction fetcher not having anywhere to place any more of them.
In this particular case here, we are far from reaching such a bottleneck.

In conclusion, what we've observed in the Timeline diagram is a clear win here.


III. Assembly instructions emitted by the compiler.

Total instructions emitted for the inner loop that is >99% of the work:

Non-interleaved: 40 instructions PER ITERATION, 2 work units = 80 instructions.
Interleaved  x2: 73 instructions PER ITERATION, 2 work units = 73 instructions.

Same amount of work in fewer assembly instructions. Definitely a win.

Instructions that have a latency of 0 cycles (handled by the Register Renamer
as simply changing a pointer to different entries on the physical register file
OR subjected to instruction fusion or micro-op fusion) closer to each other,
possibly alleviating the CPU backend's ports:

Non-interleaved: 2 instructions with a latency of 0 cycles PER ITERATION.
Interleaved  x2: 4 instructions with a latency of 0 cycles PER ITERATION.

The fact that in the interleaved version we've done 2 units of work in 73
instructions, of which 4 had a latency of 0, while the non-interleaved version
does 2 units of work in 80 instructions, 4 of which have a latency of 0, COULD
indicate that the CPU backend's ports are more alleviated per 2 units of work
(in this case - 2 processed BigInt 64-bit limbs) in the interleaved version.

Over time this could snowball into a significant win for port utilization.
While the non-interleaved version also has 4 instructions with 0 cycle latency
per 2 units of work, these 4 instructions are spread over more instructions
in total, which could be a factor in speeding up the interleaved version.


Conclusion: Interleaving 2 Montgomery MUL streams successfully sped things up
            when doing two Montgomery Power operations back to back for
						Schnorr Signature validation.

            My hypothesis as to why it was faster: The out-of-order semantics
						in modern CPUs, in my particular case, allowed the backend's
						execution ports to NOT sit idle, like they did with non-interleaved
						version, but instead to pick up out-of-order work from the second
						Montgomery multiplication stream's micro-ops that were already
						waiting in the Reservation Station.

						In the non-interleaved version, the RS also had micro-ops ready to
						feed the execution ports, but they weren't independent, they were
						needing results from currently executing micro-ops in busy ports
						while other ports sat idle.

						My hypothesis as to why it was faster, from a CPU microarchitecture
						point of view, was validated by LLVM Machine Code Analyzer running
						a detailed CPU pipeline simulation, and by the Top-down performance
						analysis methodology.


Future optimizations:

Currently the compiler is spilling general-purpose registers to L1 cache
(stack). Investigate whether spilling them to vector registers would
speed things up even more. Be careful of which uOP execution ports are
used to handle vector register to general purpose register moves, not to
stall ports that are already the most utilized.

There are current efforts in LLVM and GCC to add optimization semantics
that will automatically spill to vector registers, but it's not there yet
from what I could gather.

Verify that the 11% core-bound L2 TMA bucket for original non-interleaved
Montgomery_MUL really is caused by the CPU execution ports not being able to
be fed micro-ops due to a loop-carried dependency chain, ie the reservation
station (micro-op scheduler) had micro-ops ready but unable to give them to
some execution ports cuz they were waiting on the result of the dep. chain.

Try the following optimization ideas for the interleaved Montgomery_MUL:

- Get rid of pointer to 3-limb variable T, as it's stored inside R's buffer.
  Each active pointer consumes a register. x64 only has 16 of them.

- Don't pass N as an arg, as it's constant and the same for both MUL streams.

- Factor out inner loop into a function that takes ->bits pointers directly
  as __restrict__ for both MUL streams, takes the two streams' q (1-limb var.
	that remains constant for the inner loop) as const u64 arguments AND the
	inner loop interleaves the code for the 2 MUL streams by doing iteration I
	for stream 1 and then REUSING FOR STREAM 2 ANY REGISTER-SIZED TEMPORARIES
	that do not matter for iteration I+1. These are at least the following:
	Uh, Ul, Vh, Vl, carryC, carryD, W. Don't have a "version2" variable of them
	for the 2nd MUL stream, simply reuse the same variables to perform iteration I
	for the 2nd MUL stream too, as they are overwritten at the start of
	iteration I+1 anyway.

RESULT: It works like a charm.
        Non-interleaved went from 5200 to 4703 microseconds.
				Interleaved     went from 5000 to 4213 microseconds.

IMPORTANT:

llvm-mca is a great tool for CPU pipeline simulation cycle by cycle. It stands
for LLVM Machine Code Analyzer. It takes an assembly code snippet and gives:

- How many execution ports the CPU backend has and which is used for what.
- which execution ports each instruction's micro-ops can go to;
- how many micro-ops each instruction is decomposed into;
- latency of each instruction;
- Reciprocal throughput of each instruction (RThroughput).
  For example, Rthroughput = 0.20 means there are enough execution ports in the
	CPU backend to be executing 5 of that instruction in parallel.
- Total cycles to run your assembly code. For all interations, if loop.
- Total number of instructions and uOPs executed. For all iterations, if loop.

Direct output dump:

Iterations:        48
Instructions:      1920
Total Cycles:      532
Total uOps:        2736

Dispatch Width:    6
uOps Per Cycle:    5.14
IPC:               3.61
Block RThroughput: 9.5

Explanation: It took a total of 1920 instructions and 2736 micro-ops to be
             executed by the CPU's pipeline to run 48 iterations in a loop
						 of the assembly code in the input file. It took 532 CPU cycles.

						 6 is the maximum number of uOPs that can be moved from the
						 CPU Frontend to the reservation station and ROB, OR from the
						 reservation station to the execution ports, depending on who
						 uses the term. Issue Width is used interchangeably sometimes.
						 Must clarify what it means in LLVM-MCA.

						 Our actual code was running on average 5.14 uOPs per cycle,
						 given the maximum of 6. This is a high utilization %, good.

						 3.61 is how many instructions are retired per cycle, on average.
						 IPC is to be maximized for high performance optimizations.

						 9.5 cycles is the theoretical minimum number of cycles it can take
						 to run the assembly code in the input file, given the specific
						 CPU microarchitecture the simulation was told to run on.

Here, we have instructions with 0 uOPs and instructions that take 0 cycles.
Explanation:

These are moves that were handled by the register renamer in the CPU frontend.
The CPU may only have 16 general purpose registers, but it has a much larger
physical file of actual hardware registers. This is how 2 instructions that
use RAX can run in parallel if the renamer assigns them different physical
registers in the register file. The Reservation Station (Scheduler) already
has info on which micro-ops need which physical registers in the file, this
is how it can feed the backend execution ports two instructions using RAX at
once. Then, because in the assembly code they were separate and came one
after another, this will be reflected in the Reorder Buffer (ROB) and this
hardware unit will make sure the instructions are RETIRED in proper order.

Another reason an assembly instruction can be shown to have 0 uOPs is because
the Decoder in the CPU Frontend fused it with the next instruction. Example:

[1]    [2]    [3]    [4]    [5]    [6]    Instructions:
 0      1     0.00                        add   rdi, 1
 1      1     0.20                        cmp   rdi, 48

The numbers here are for the entire run, on average, for the code we executed.
The first mov instruction went to EITHER ONE of ports [2] [3] [4]. A third of
the times it had to be run in the backend, it used port 2 for its one uOP.
A third of the time, it used port 3 and a third of the time it used port 4.
Only one of these ports is needed for it. In contrast, for the MULX instruction
the table shows it has a total of 2.0 micro-ops, and each micro-op MUST GO
to port [1] and to port [5] - the instruction uses them in parallel when both
ports are available and takes 4 cycles to finish. This means, each time this
instruction (MULX) was fed to the backend, one of its micro-ops used port 1 and
one of its micro-ops used port 5 and the 1.00 on both ports here means 100%
of the times the instruction had to be executed, it used this port.

Something like:
0.04    -      -      -      -      -     1.96    -      -     setb r15b

means the instruction has 2 uOPs and 96% of the time the instruction had to
run in the backend, both of these uOPs used port 6 one after another. In 4% of
the time, it could use port 0 for one of the micro-ops, while port 6 runs the
other micro-op, both in parallel.

HOW TO READ THIS:

In cycle 0, 5 instructions were dispatched from the frontend INTO the
Reservation Station. In the next cycle, the reservation station managed to
place the first 2 instructions' uOPs into execution ports. In cycle 7 the first
2 instructions retired. First instruction was executing in cycles 1, 2, 3, 4, 5
and in cycle 6, its result was ready and visible to the CPU pipeline. In cycle 7
it entirely left the backend and was retired in the same cycle as instruction 2.
Instructions 2 and 3 had a data dependency on previous instructions & could NOT
start to be executed on the backend ports. In cycle 6 and 7 they started being
processed in the execution ports, respectively, marked by the lower-case e.

Instruction [0,8] aka the 9th instruction of the first iteration of the assembly
code finished its work in cycle 8, but had to WAIT for previous assembly
instructions TO BE READY TO BE RETIRED, before IT can retire, marked by E------
and same for instruction [0,9] and [0,11]. All of them retired in cycle 24.

   -------------------------------------------------------------------------

Profiling harness automated to run Simulation 1 128 times for 128 averages.
Each average is made up of 500 completely organic calls of the montgomery
powering function over 30 seconds. The measurements.txt file shows each
stabilized average (from 500 calls) with outliers properly excluded via first
and third quartile statistical method:

1. Sort the data points in ascending order. Split into 2 halves.
2. middle-indexed value of first half (aka median) = Q1.
3. middle-indexed value of other half (aka median) = Q3.
4. Interquartile range IQR = Q3 - Q1.
5. Lower fence LF = Q1 - (1.5 * IQR).
6. Upper fence UF = Q3 + (1.5 * IQR).
7. Outliers are all data points X < LowerFence  OR  X > UpperFence.

Outliers are calculated for and removed from the 500 function call latencies
that were used to produce each of the 128 stabilized averages over 30 seconds.
Outliers are not removed from the 128 stabilized averages themselves. It takes
the automated latency measurement harness a little over 1 hour to collect all
128 stabilized averages and give the average for a SET.

Lastly, a final average is calculated of the 128 stabilized averages.
These are the results displayed here.

Each set takes ~ 1 hour to obtain.

Huge compiler optimization success story - factoring out the loop in ChaCha20
that writes out the final result blocks into its own function, passing pointer
arguments as __restrict__ (because that's guaranteed by design here) actually
led the compiler to stop emitting a slow and a fast vectorized version of the
loop (explicitly stating that it's doing it out of fear for possible aliasing)
and it started emitting only the vectorized loop version. Interestingly, opening
a sub-block in the function wherein to have the code for this loop and declare
separate pointers to the ones used in the loop to iterate over the arrays and
declare THOSE scope-local pointers as __restrict__ DID NOT cause the compiler to
drop the 2 loop versions. It appears that __restrict__ works best to convince
the compiler of our promise when used on function arguments.
