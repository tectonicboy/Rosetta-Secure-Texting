May 11-13, 2026 |

Verify that the 11% core-bound L2 TMA bucket for original non-interleaved
Montgomery_MUL really is caused by the CPU execution ports not being able to
be fed micro-ops due to a loop-carried dependency chain, ie the reservation
station (micro-op scheduler) had micro-ops ready but unable to give them to
some execution ports cuz they were waiting on the result of the dep. chain.

Try the following optimization ideas for the interleaved Montgomery_MUL:

- Get rid of pointer to 3-limb variable T, as it's stored inside R's buffer.
  Each active pointer consumes a register. x64 only has 16 of them.

- Don't pass N as an arg, as it's constant and the same for both MUL streams.

- Factor out inner loop into a function that takes ->bits pointers directly
  as __restrict__ for both MUL streams, takes the two streams' q (1-limb var.
	that remains constant for the inner loop) as const u64 arguments AND the
	inner loop interleaves the code for the 2 MUL streams by doing iteration I
	for stream 1 and then REUSING FOR STREAM 2 ANY REGISTER-SIZED TEMPORARIES
	that do not matter for iteration I+1. These are at least the following:
	Uh, Ul, Vh, Vl, carryC, carryD, W. Don't have a "version2" variable of them
	for the 2nd MUL stream, simply reuse the same variables to perform iteration I
	for the 2nd MUL stream too, as they are overwritten at the start of
	iteration I+1 anyway.

RESULT: It works like a charm.
        Non-interleaved went from 5200 to 4703 microseconds.
				Interleaved     went from 5000 to 4213 microseconds.

Profiling harness automated to run Simulation 1 128 times for 128 averages.
Each average is made up of 500 completely organic calls of the montgomery
powering function over 30 seconds. The measurements.txt file shows each
stabilized average (from 500 calls) with outliers properly excluded via first
and third quartile statistical method:

1. Sort the data points in ascending order. Split into 2 halves.
2. middle-indexed value of first half (aka median) = Q1.
3. middle-indexed value of other half (aka median) = Q3.
4. Interquartile range IQR = Q3 - Q1.
5. Lower fence LF = Q1 - (1.5 * IQR).
6. Upper fence UF = Q3 + (1.5 * IQR).
7. Outliers are all data points X < LowerFence  OR  X > UpperFence.

Outliers are calculated for and removed from the 500 function call latencies
that were used to produce each of the 128 stabilized averages over 30 seconds.
Outliers are not removed from the 128 stabilized averages themselves. It takes
the automated latency measurement harness a little over 1 hour to collect all
128 stabilized averages and give the average for a SET.

Lastly, a final average is calculated of the 128 stabilized averages.
These are the results displayed here.

Each set takes ~ 1 hour to obtain.

Huge compiler optimization success story - factoring out the loop in ChaCha20
that writes out the final result blocks into its own function, passing pointer
arguments as __restrict__ (because that's guaranteed by design here) actually
led the compiler to stop emitting a slow and a fast vectorized version of the
loop (explicitly stating that it's doing it out of fear for possible aliasing)
and it started emitting only the vectorized loop version. Interestingly, opening
a sub-block in the function wherein to have the code for this loop and declare
separate pointers to the ones used in the loop to iterate over the arrays and
declare THOSE scope-local pointers as __restrict__ DID NOT cause the compiler to
drop the 2 loop versions. It appears that __restrict__ works best to convince
the compiler of our promise when used on function arguments.
